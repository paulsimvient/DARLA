1. Executive Summary

DARLA is a deterministic, headless C++20 simulation that models a maritime intelligence, surveillance, and reconnaissance (ISR) mission degraded by cyber attack. The system is built to answer a specific class of operational-science questions:

When sensing or communications fail mid-mission, which intervention is causally justified, under what authority, and can that claim survive executable falsification?

Primary innovation: DARLA’s contribution is not that agents make recommendations, but that each recommendation is linked to evidence, causal parents, counterfactual branch tests, authority state, model validity, and replayable falsification outcomes. Unsupported recommendations are downgraded or refused rather than converted into narrative explanations. The system is designed to say no.

Unlike a traditional wargame or game engine, DARLA treats the simulation as an evidence-producing causal experimentation instrument, not a richer battlespace visualization. Every state transition is recorded in an append-only event ledger. Causal relationships are represented in a temporal causal graph and a typed relationship graph loaded from scenario YAML. Online agents observe degraded state, query bounded counterfactual branches, propose courses of action (COAs), and—when authorized—schedule interventions. A separate slow loop replays and falsifies those decisions asynchronously, scoring them against planted causal truth and credibility contracts.

A React dashboard provides operational visualization (live tick streaming, map, COA lifecycle, claims panel). It is a consumer of kernel outputs—not the source of simulation truth. Unreal, full LVC adapters, and LLM agents remain out of scope.

The C++ kernel owns truth: state, events, replay hash, counterfactual branches, and credibility assessments.

2. Problem Domain

2.1 Operational narrative

Blue forces operate a UAS over the Taiwan Strait micro-world. A red cyber actor degrades blue sensing and/or communications. Mission success requires detecting a maritime target before T+1800 with sufficient track confidence and engagement authority.

The commander must decide among interventions such as:

  • isolate_compromised_sensor_feed
  • restore_comms_relay
  • enable_autonomous_search
  • pre_authorize_engagement
  • remove_logistics_delay (confounder)

Wrong choices waste tempo; late choices cannot rewrite prior detection outcomes. Logistics delay is correlated with failure but is not the root cause—a deliberate confounder for falsification testing.

2.2 Scientific narrative

The system must support defensible causal experimentation:

  1. Observe degradation through agent beliefs and events
  2. Estimate action effects via counterfactual branches, not heuristics alone
  3. Decide under explicit authorization policy
  4. Validate decisions through replay, falsification, and planted-truth scoring
  5. Replay deterministically for audit and hash verification

4. The Simulation Kernel

The kernel is the single source of simulation truth.

// Each tick:
resetTickBudgetUsage(world.runtime_budgets);
MicroWorldAdjudicator::adjudicate(world, ledger, causal_graph);
++world.tick;

Initialization loads scenario YAML: entities (commander, UAS, relay, cyber actors, target), mission constraints, scripted events, relationship edges, and planted causal truth.

Determinism is enforced via seeded RNG (std::mt19937_64), snapshot/restore for branch exploration, and a replay hash over the event ledger.

Snapshots capture (WorldState, EventLedger, TemporalCausalGraph) and enable counterfactual branches without contaminating the main timeline until an intervention is committed.

5. Per-Tick Adjudication Flow

Each simulation tick follows a fixed adjudication pipeline:

1. Reset per-tick execution budgets
2. Pre-cyber phase     → RedCyberAgent (attack/degrade)
3. Cyber adjudication  → apply degradation to world components
4. Post-cyber phase    → Sensor, Comms, UAS, Logistics,
                          CausalMonitor, Credibility, Commander
5. Advance tick counter

Agents do not freely mutate the world every tick. They observe, monitor, detect anomalies, recommend COAs, and act once when causal evidence and authority permit. This “always-on but restrained” model (Phase D) replaces one-shot agent flags with persistent belief state.

The current prototype implements a heterogeneous agent framework with commander, sensor, comms, UAS, logistics, causal-monitor, and credibility roles. Not all roles are equally autonomous: sensor, comms, UAS, and logistics agents primarily operate as monitors and structured event emitters; the causal monitor and credibility agents scan validity envelopes; only the BlueCommanderAgent performs bounded COA orchestration under authority policy. Red cyber behavior remains scripted, not adaptive.

6. Event Ledger and Causal Evidence

Every meaningful observation, policy proposal, intervention, and budget violation becomes a SimEvent:

  • Monotonic event_id
  • Simulation tick
  • Actor entity
  • Typed EventType and string label (e.g. coa_recommendation, sensor_confidence_loss)
  • State deltas (before/after field changes)
  • causal_parent_events linking evidence chains
  • Provenance, confidence, uncertainty

The ledger is the audit trail. The dashboard timeline, causal claims, and replay verification all derive from it.

The temporal causal graph accumulates edges between events and state variables over time. The relationship graph (from YAML) encodes structural dependencies: commands, senses, degrades, communicates_with, supplies, etc.

7. Dual-Loop Runtime

DARLA separates operational decision speed from scientific validation depth.

7.1 Fast loop (online, per tick)

  1. Agents update beliefs (sensor trust, comms health, mission risk, COA entropy)
  2. Commander reviews COAs on a 60-tick cadence or on anomaly edge
  3. For each candidate action, CausalActionEstimator runs bounded counterfactual branches
  4. Commander ranks candidates by score (mission gain, confidence, cost, risk, uncertainty)
  5. AuthorityGate evaluates whether execution is permitted
  6. Approved action is scheduled at a future tick via InterventionEngine

7.2 Slow loop (async, post-decision)

After the commander’s primary decision, AsyncValidationRuntime queues validation jobs that:

  • Replay and falsify the online causal assumption
  • Run minimum intervention search
  • Score recovery against planted causal truth edges from scenario YAML
  • Produce an AsyncValidationSummary (survived/overturned, best interventions, recovery score)

The slow loop is budget-limited (async_replay_jobs, async_branch_executions per tick) so real-time stepping remains bounded.

8. Structural Causal Layer (Phase A)

Prior to Phase A, commander logic leaned on hard-coded scores. The current design uses:

8.1 RelationshipGraph

Typed edges from scenario YAML connect agents and entities. Example:

relationships:
  - type: commands
    source: blue_commander
    target: blue_uas_1
  - type: degrades
    source: red_cyber_actor
    target: blue_uas_1
    component: sensor

The commander discovers intervention targets through graph traversal (GraphEntityResolver), reducing hard-coded entity names.

8.2 CausalActionEstimator

For each candidate action, the estimator:

  1. Takes a branch snapshot at the current tick
  2. Projects baseline mission metrics over a decision horizon
  3. Runs InterventionEngine counterfactual branches
  4. Returns ActionEffectEstimate: expected mission gain, causal confidence, cost, risk, support flag, structured rationale

Unsupported actions are refused. The commander holds COA when evidence is weak.

8.3 InterventionEngine

Interventions are time-faithful: scheduled at --at tick and applied only when simulation time reaches that tick. Early intervention can recover the branch; late intervention cannot undo prior detection or mission failure.

9. Continuous COA Orchestration (Phase E)

Phase E elevates the commander from a one-shot decision to continuous course-of-action orchestration.

9.1 COA model

Each CourseOfAction carries:

  • Action type, target, proposed tick, scheduled execution tick
  • Expected mission gain, causal confidence, cost, risk, composite score
  • Rationale and lifecycle status: Proposed → Recommended → Approved → Executing → Completed (or Rejected / Superseded)

9.2 Review cadence

  • Periodic review: every 60 ticks
  • Anomaly-triggered review: when sensor/comms anomaly edges fire (without re-firing every tick while anomaly persists)

Each review emits a ranked coa_recommendation event with causal parent links to the triggering observations.

9.3 Authorization modes

ModeBehavior
PolicyAutoCommander executes best supported action when authority permits
ExplicitApprovalsOnly pre-approved action@tick tokens execute
HumanHoldRecommendations only; no autonomous execution

The dashboard COA panel exposes mode selection and approve/reject controls. Re-running with --approve isolate_compromised_sensor_feed@760 drives explicit approval flows.

9.4 COA entropy

The commander belief state tracks COA entropy—a measure of dispersion across ranked recommendations—surfaced in emergence detection and live frames for operational uncertainty signaling.

10. Scientific Rigor Layer (Phase C)

10.1 Typed causal variables

Claims reference structured variables (e.g. blue_uas_1.sensor.confidence, detection_time, mission_success_score) rather than event labels alone.

Typed causal variables reduce dependence on string labels, but the current micro-world still relies on scenario-provided relationship structure, planted truth edges, and label-assisted claim construction in CausalClaimBuilder. Fully autonomous causal discovery across arbitrary LVC scenarios remains future work.

10.2 Credibility contracts

YAML models under models/ declare intended use, validity envelope, assumptions, and known failure modes. CredibilityEngine combines evidence strength, model risk (decision influence × consequence of error), falsification outcome, and contract compliance into a CredibilityAssessment with a reportable boolean gate.

Example contract excerpt (models/sensor-confidence-v0.yaml):

model_name: UAS sensor confidence model
intended_use: causal explanation over deterministic UAS maritime ISR micro-world
valid_conditions:
  - maritime ISR
  - modeled RF/cyber degradation
invalid_conditions:
  - dense urban terrain
  - unmodeled space ISR
known_failure_modes:
  - adversary spoofing not represented

A claim is reportable only when ledger evidence exists, executable falsification survives, and credibility_score ≥ 0.65. Correlation-only claims (e.g. logistics delay) are explicitly marked rejected as causal under executable falsification branches while still appearing in the assessment output—the system documents why they were refused.

10.4 Planted causal truth

Scenario YAML declares ground-truth causal edges. After a run, PlantedTruthScorer compares recovered claims to planted edges. Current metrics (all computed in PlantedTruthScore):

  • Edge recall / recovery score: matched_edges / planted_edges (unit test threshold ≥ 0.66 for 3-edge scenario)
  • Edge precision proxy: extra_edges lists recovered-but-unplanted claims
  • Pathway coverage: matched_edges and missing_edges per planted edge
  • False-cause rejection: logistics confounder overturned via falsification branches (not counted as planted match)
  • Minimum intervention recovery: async slow loop reports lowest-cost and best-effect interventions alongside recovery score
planted_causal_truth:
  edges:
    - red_cyber_actor.degrade_sensor_feed -> blue_uas_1.sensor_confidence_loss
    - blue_uas_1.sensor_confidence_loss -> delayed_target_detection

10.5 Emergence detection

EmergenceDetector identifies metric-based patterns: tempo collapse, sensor trust collapse, comms cascade, COA entropy spikes—without requiring pre-scripted labels for every failure mode.

10.6 Minimum intervention search

sim-search brute-forces singleton and pair interventions, reporting lowest-cost effective and best mission-effect combinations. In the default micro-world, sensor isolation is typically the lowest-cost effective singleton; paired actions can improve detection time further.

11. Execution Budgets and Live Runtime (Phase F)

Real-time operation requires bounded work per tick. ExecutionBudget enforces:

BudgetDefault limitConsumer
Agent decisions16/tickAgent orchestrator
Causal queries12/tickCommander estimator calls
Async replay jobs2/tickSlow loop
Async branch executions24/tickFalsification probes

Exceeding a budget emits budget_exceeded events rather than silently running unbounded counterfactual exploration—preserving determinism and predictable latency.

Budget semantics in the runtime:

ConceptWhere it lives
Configured budgetExecutionBudgetLimits on WorldState.runtime_budgets
Observed usage (per tick)tick_usage — reset each kernel step; surfaced in live SSE frames
Observed usage (cumulative)total_usage — accumulated across the run
Budget violation eventbudget_exceeded ledger event with consumer label
Latency impactCommander returns supported=false with rationale “causal query budget exceeded”; async validation aborts with budget summary

Acceptance test: unit tests force causal_queries_per_tick = 1, run commander estimation, and require a budget_exceeded event while deterministic replay hash remains stable. Budget compliance is measured behavior, not configuration text alone.

11.1 sim-live SSE stream

sim-live steps the kernel tick-by-tick and emits Server-Sent Events:

EventPayload
metaScenario id, seed, authorization mode, max ticks, mission cutoff
tickFrame: metrics, entities, agent beliefs, budgets, tick events
doneFinal tick and mission metrics

The Node server proxies this at /api/live. The dashboard “Live tick stream” mode (default on) connects via EventSource, updating the map and metrics as ticks arrive.

11.2 Dual data path in the dashboard

Live mode runs two parallel pipelines:

  1. Live path: sim-live → SSE → incremental playback frames (real-time map/beliefs/tick counter)
  2. Analysis path: sim-export → full dashboard JSON (claims, planted truth, async validation, COA log, intervention search)

Export is batch-oriented and slower (~10–20s). The UI waits for at least one live frame before rendering playback-dependent panels, avoiding crashes when export completes before the first SSE tick.

Batch mode (live stream unchecked) uses sim-stream for full post-run playback JSON instead.

13. Dashboard Behavior

The dashboard is an evidence consumer, not the simulation kernel. It helps operators and reviewers inspect what the C++ runtime already decided and recorded. Kernel behavior is identical whether or not the UI is open.

When you click Run Simulation:

  1. API server spawns C++ binaries from build-make/ or build/
  2. Live mode: EventSource opens; meta sets playback metadata; each tick appends a frame
  3. In parallel: sim-export produces the full analysis payload
  4. UI shows streaming banner until frames arrive; then renders:
    • Live tick metrics and MapLibre operational map
    • Playback bar (scrub/play/pause, decision markers at COA events)
    • COA panel with authority mode and approve/reject
    • Dual-loop panel (fast decision vs slow validation summary)
    • Causal claims, planted truth, intervention search, timeline, COA lifecycle log

Deterministic replay hash is displayed as a ledger integrity fingerprint.

Run locally:

cmake --build build-make
./restart-dashboard.sh
# UI: http://localhost:5176
# API: http://localhost:8787

15. Reference Scenario Structure

The UAS maritime cyber scenario defines:

  • Entities: commander, UAS, comms relay, cyber defense, red cyber actor, maritime target, logistics node
  • Components: kinematic (map coordinates), sensor, comms, cyber
  • Mission: detect target before tick 1800, minimum track confidence, engagement authority
  • Scripted events: cyber degradation timing, logistics delay confounder
  • Relationships: command, sensing, degradation, communication, supply edges
  • Planted truth: causal edges for scoring

A comms-only variant (comms-only.yaml) isolates comms degradation with a healthy sensor—restoring comms alone recovers mission timing; sensor isolation is the wrong response. This validates that the estimator and commander discriminate failure modes.

17. Known Boundaries and Non-Goals

The prototype is explicit about what it does not yet claim:

  • No adaptive adversary beyond scripted red cyber behavior
  • Causal discovery is partially label-assisted and scenario-seeded—not fully unsupervised across arbitrary domains
  • No Unreal/visual LVC, no LLM agents, no production VV&A suite
  • Planted-truth scoring uses edge recall/recovery today; full precision/F1 pathway taxonomy is not yet a first-class API
  • Claim status taxonomy in Section 10.3 mixes target vocabulary with current CredibilityAssessment fields
  • Live SSE frames are compact; full COA/authority fields may differ from batch sim-stream richness
  • Dashboard is visualization only—never authoritative over the C++ ledger

Remaining DARPA-hard gaps: full adversarial adaptation, autonomous causal discovery at scale, and enterprise-grade validation tooling.

18. Conclusion

DARLA implements a causal-agentic simulation stack—demonstrated today in the UAS maritime cyber micro-world—where:

  1. A deterministic kernel produces auditable event evidence
  2. A heterogeneous agent framework combines monitors, validity scanners, and a single authority-gated commander under per-tick budgets
  3. COA orchestration uses graph-native causal estimation and explicit authorization policy
  4. A slow loop falsifies online decisions against counterfactual branches and planted truth
  5. A dashboard consumes kernel outputs for operational inspection—not as a second source of truth

The central thesis: real-time agentic decisions are allowed, but causal truth is earned through deterministic evidence, replay, falsification, planted-truth scoring, and credibility envelopes—not through richer visualization or narrative explanation alone.

When you run the dashboard, watch ticks stream to T+1440, approve a COA, and read falsification survive or overturn in the dual-loop panel, you are observing operational tempo and causal rigor in the same deterministic runtime—with pass/fail acceptance criteria enforced in CI, not asserted in prose alone.
